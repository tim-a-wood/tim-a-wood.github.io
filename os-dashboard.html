Top priority — Cybersecurity or Development should own fixes
Security
A tampered project file could run unwanted scripts inside the room editor
Some room and door names are dropped straight into the page. If someone opens a crafted file, the browser might run code. Cybersecurity should review; Development should use the same safe text handling the editor already uses elsewhere.
Reliability and keyboard use
A text color setting points at a theme name that does not exist
In the sprite workbench marketing area, a style asks for a color variable that was never defined. The browser may skip the style, so text can look wrong or invisible. Design and Development should align it with the official color list.
Keyboard users cannot see which control is selected in the room editor
Native buttons and fields need a clear focus ring for people who tab through the UI. This is a basic accessibility expectation. Development can fix with one shared focus style.
Next priority — Development and Design should tighten consistency
Design system drift
Corner rounding on the sprite workbench does not match the house style
Small and medium rounded corners use different numbers than the rest of the product, so the workbench can look subtly “off” next to the room editor.
Many font sizes are set in a flexible unit instead of the shared size scale
Several screens mix rem-based sizes with the fixed scale the style guide defines. That makes future redesigns harder and can hurt readability.
Several screens still use one-off hex colors
A handful of tints are hard-coded instead of pulled from the shared palette, so dark mode and future theme work stay inconsistent.
One shadow style name is not on the official list
A shortcut shadow name appears in the sprite shell but is not documented with the other shadows. Replace with the standard large shadow so tooling and docs stay in sync.
Motion and spacing
Some hover and open animations use timing outside the agreed range
A few panels animate slower or faster than the rest of the product. Standardize on the short and medium durations from the style guide.
Some rounded corners use values that are not on the approved list
Mix of circles, pills, and ad-hoc pixel radii. Map each to the nearest approved radius so everything feels like one product.
Gaps and padding sometimes land between the 4-pixel grid
Room wizard and sprite shell both have a few odd pixel values. Nudging to the grid keeps layouts crisp and matches the rest of the tools.
A few type sizes are smaller than the minimum or not on the scale
The room editor shows some labels at sizes below the readable minimum or outside the allowed set. Bump to the nearest approved size.
Opportunities
High payoff
Put shared colors and type rules in one file every tool imports
Stops the same token block being edited in three places and drifting apart.
Split the giant room editor page into smaller script files
The file is enormous; breaking it up makes security review and onboarding easier.
Add the shared keyboard focus style to the room editor
Small change, big win for people who do not use a mouse.
Medium payoff
One shared stylesheet for the top navigation
Room editor and sprite workbench repeat the same nav styling; one source of truth reduces drift.
Bring the marketing section of the sprite shell onto the official palette
Replace leftover custom colors and sizes with tokens from the style guide.
Match sprite workbench corner radii to the canonical scale
Aligns the workbench with room editor and dashboard chrome.
