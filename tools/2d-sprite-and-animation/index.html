What it is

Sprite Workbench is built for a specific job: carrying one side-view character from first brief to runtime handoff without losing the thread.

The workflow stays narrow on purpose. Describe the character. Choose the look. Build the motion. Export the package.

Where it fits best.

Best when the job is specific and the handoff matters.

Best for

  • Solo devs and small teams building side-view 2D games
  • Projects that need a direct line from character concept to runtime assets
  • Teams that want local project ownership instead of cloud-only state

Not for

  • Broad concept exploration across many styles
  • Front-view, isometric, or full-sheet character pipelines
  • Large collaboration suites or marketplace workflows

Quickstart

  1. Open Sprite Creation and create a project.
  2. Describe the character in plain language and save the setup.
  3. Create or import concepts, then lock one side-view source.
  4. Generate motion clips in the Animations step.
  5. Run checks and export the runtime package in Review & Export.
The goal is a side-view character package you can carry straight into runtime.

Workflow overview

  • Describe: capture role, silhouette, materials, prop, and tone.
  • Concepts: build or import options, then lock the side-view source you want to keep.
  • Animations: build named motion clips from that locked source.
  • Review & Export: check the package and export the runtime handoff.

Export format

The workbench keeps project backup and runtime export separate.

  • Project export: a restorable bundle for backup and transfer.
  • Runtime export: the package the game consumes.

The current runtime package includes:

  • animation_sheets/<clip>.png
  • animation_sheets/<clip>.json
  • atlas.json and animations.json
  • Preview GIFs for quick review
  • Export manifest and verification output

Phaser guide

Ashen Hollow is the reference consumer for the current runtime package.

  • The checked-in sample handoff lives at assets/test-hero/runtime-manifest.json.
  • The sample frame payload lives at assets/test-hero-frames/.
  • Map locomotion states to clips like idle, run, and jump.
  • Treat action states like attack and parry as deliberate runtime triggers.
  • Plan to tune scale, origin, and collision fit once the sprite is in-engine.

FAQ

Who is this for?
Small teams and solo devs building side-view 2D characters.

Why local-first?
So projects can be restored, moved, and backed up without treating the browser as the only place the work exists.

Why separate Export Project from Export Package?
One preserves the editable project. The other produces the runtime handoff.

Is this meant to be a general-purpose sprite editor?
No. It is tuned for a narrower job than that.
