# Production-Ready Concept-To-Sprite Specification

## Purpose

This document defines the implementation plan for taking the current sprite workbench from:

- AI concept generation + placeholder downstream sprite production

to:

- AI concept generation + real concept-to-sprite conversion + deterministic game-ready sprite output

This is an implementation handoff for an AI agent.

This document is authoritative for the production-ready sprite pipeline.

## Core Product Contract

The tool must support this end-to-end workflow:

1. Generate concept art from a prompt and optional references.
2. Approve one concept.
3. Convert that approved concept into a real canonical sprite model.
4. Use that canonical sprite model to render final animation frames deterministically.
5. Export a game-ready sprite package.

`Game-ready` means:

- transparent background
- fixed frame size
- fixed pivot
- stable anatomy
- stable draw order
- no per-frame AI hallucination
- exact metadata for runtime use

The final animation frames must not be generated by freeform AI image generation.

## Current Problem To Replace

The current tool generates real concept boards, but the approved concept is not actually turned into the final sprite art.

The current downstream path:

- builds synthetic parts from a hardcoded `PART_LIBRARY`
- rigs those synthetic parts
- renders deterministic idle/walk frames from those synthetic parts

This means the current output is deterministic but not character-specific.

The entire point of this spec is to replace that placeholder downstream path with a real production path derived from the approved concept.

## Non-Negotiable Design Rule

Use AI only where it provides leverage without introducing nondeterminism into final animation output.

Allowed AI stages:

- concept generation
- concept refinement
- master-pose generation
- optional one-off assisted occlusion recovery during part extraction

Disallowed AI stages:

- generating final animation frames
- redrawing each frame of idle/walk/jump/attack
- freeform img2img during final production rendering

Once the canonical sprite model is approved, all downstream rendering must be deterministic.

## Production Scope

### Initial Production Slice

The first production-ready slice must support exactly:

- one humanoid biped
- strict side view
- one facing direction
- one handheld prop maximum
- stylized raster sprite output
- final frame size `256x256`
- `idle`
- `walk`

### Second Production Slice

After the first slice is stable, expand to:

- `jump`
- `attack`

Do not start with 4-direction, 8-direction, quadrupeds, capes, wings, or multi-character scenes.

## Required New Pipeline

The real pipeline must be:

1. `Brief`
2. `Concept Board`
3. `Concept Approval`
4. `Master Pose`
5. `Sprite Model Build`
6. `Sprite Model Correction`
7. `Rig`
8. `Animation Clip Render`
9. `QA`
10. `Export`

This replaces the current fake path:

- approved concept -> synthetic part library -> fake rig -> render

with:

- approved concept -> approved master pose -> extracted sprite model -> deterministic rig render

## Stage Definitions

### 1. Brief

Purpose:

- collect a production-aware character description
- collect optional references
- normalize prompt intent toward Ashen Hollow house style and sprite readability

Output:

- `brief.json`

Requirements:

- keep current structured brief behavior
- bias prompt defaults toward Ashen Hollow dark-fantasy side-view readability
- preserve plain-language wizard terminology

### 2. Concept Board

Purpose:

- generate 6 initial concept looks
- generate 4 refinement looks

Output:

- concept images
- prompt lineage
- concept metadata

Requirements:

- keep current ComfyUI concept backend
- concepts must remain AI-generated
- concept review stays in the wizard/workbench

### 3. Concept Approval

Purpose:

- freeze one concept as the style and identity source

Output:

- `character_spec.json`

Requirements:

- store chosen concept id
- store prompt lineage
- store palette direction
- store locked attributes

### 4. Master Pose

Purpose:

- produce the final approved source image used to build the sprite model

Definition:

- one clean full-body side-view image
- plain or removable background
- no environment scene
- readable silhouette
- correct orientation for sprite extraction

Rules:

- generate exactly 3 master-pose candidates from the approved concept
- allow optional references and the approved concept to condition this step
- user must choose one
- this is the last required AI image generation step before deterministic production

Output:

- `master_pose/master_pose_01.png`
- `master_pose/master_pose_02.png`
- `master_pose/master_pose_03.png`
- `master_pose/approved_master_pose.png`
- `master_pose/master_pose_manifest.json`

### 5. Sprite Model Build

Purpose:

- convert the approved master pose into a canonical layered sprite model

Definition:

The `SpriteModel` is the deterministic source of truth for final sprite rendering.

Required parts:

- `head`
- `hair_front`
- `hair_back`
- `torso`
- `pelvis`
- `upper_arm_left`
- `lower_arm_left`
- `hand_left`
- `upper_arm_right`
- `lower_arm_right`
- `hand_right`
- `upper_leg_left`
- `lower_leg_left`
- `foot_left`
- `upper_leg_right`
- `lower_leg_right`
- `foot_right`
- `prop`
- `accessory_front`
- `accessory_back`

Required build outputs:

- transparent PNG for each part
- mask for each part
- part pivot
- parent joint
- draw order
- palette swatch set
- outline style metadata

Allowed one-off AI assistance:

- optional hidden-area recovery for occluded parts

Constraint:

- any AI-assisted recovery must happen at part-build time only
- once accepted, the recovered part becomes frozen deterministic source art

Output:

- `parts/*.png`
- `parts/masks/*.png`
- `sprite_model.json`
- `palette.json`

### 6. Sprite Model Correction

Purpose:

- let the user or agent fix extraction errors without leaving the tool

This is mandatory. Do not rely on perfect auto-extraction.

Required deterministic editing operations:

- add to mask
- remove from mask
- split mask into named parts
- merge parts
- rename part
- set pivot
- move layer
- scale layer
- rotate layer
- set draw order
- set parent joint
- mirror left/right part
- duplicate part
- crop/pad part
- alpha cleanup
- outline normalization
- palette replace

Optional one-off assisted operation:

- `recover occluded region`

If implemented, this must:

- run only on a selected part
- save each result variant separately
- require explicit approval before it modifies the canonical part

Output:

- corrected `parts/*.png`
- corrected `sprite_model.json`
- `sprite_model_history.json`

### 7. Rig

Purpose:

- bind the extracted parts to a deterministic animation rig

Required joints:

- `root`
- `pelvis`
- `torso`
- `neck`
- `head`
- `shoulder_left`
- `elbow_left`
- `wrist_left`
- `shoulder_right`
- `elbow_right`
- `wrist_right`
- `hip_left`
- `knee_left`
- `ankle_left`
- `hip_right`
- `knee_right`
- `ankle_right`

Rig outputs:

- neutral pose render
- per-part pivot map
- joint map
- draw-order rules
- prop attachment joint
- foot anchors
- rotation limits

Constraint:

- rig must be built from the extracted sprite model, not from synthetic placeholder parts

Output:

- `rig.json`
- `rig/neutral_pose.png`
- `rig/joint_map.json`
- `rig/pivot_map.json`

### 8. Animation Clip Render

Purpose:

- render final animation frames deterministically from the rigged sprite model

Rules:

- no AI calls in this stage
- animation clips are explicit transform data
- renderer composites extracted parts using rig transforms and corrective assets

Required clips in first slice:

- `idle`
- `walk`

Required clip data:

- exact frame count
- exact fps
- exact per-frame transforms
- root motion policy
- loop continuity rules

Optional deterministic corrective assets:

- elbow bends
- knee bends
- shoulder overlap corrections
- cloth hem alternates
- prop hold alternates

These are not AI frames. They are additional canonical assets used by the renderer.

Output:

- `animations/idle/*.png`
- `animations/walk/*.png`
- `animation_clips.json`

### 9. QA

Purpose:

- prevent export when deterministic production output is invalid

Required checks:

- correct frame count
- exact frame size
- transparent background
- exact pivot
- no clipping
- no missing parts
- no duplicate prop
- stable draw order
- stable foot anchor
- loop seam continuity
- spritesheet packing correctness
- metadata correctness

Output:

- `qa_report.json`

### 10. Export

Purpose:

- package the final deterministic output for game runtime use

Required outputs:

- `spritesheet.png`
- `frames/*.png`
- `atlas.json`
- `animations.json`
- `preview.gif`
- `export_manifest.json`

## Wizard Requirements

The wizard must remain a separate guided flow for `Create New Sprite`.

The multi-panel workbench remains in the product.

The wizard must not only collect input. Each step must include the appropriate visualization so the user always sees both:

- current state
- current output

## Deterministic Tooling Requirement

The implementation must intentionally move functionality out of probabilistic AI and into deterministic tooling.

The deterministic tools must be usable by:

- the user directly
- future automation or AI-assisted orchestration

That means the canonical operations must be explicit commands on persisted assets, not hidden manual assumptions.

Required deterministic tool surfaces:

- mask editor
- part editor
- pivot editor
- draw-order editor
- palette editor
- corrective-asset editor
- clip transform editor

Do not build a system where the only way to fix a sprite is “rerun AI and hope.”

## API Changes

Add the following endpoints:

- `POST /api/projects/<project_id>/master-pose/generate`
- `POST /api/projects/<project_id>/master-pose/select`
- `POST /api/projects/<project_id>/sprite-model/build`
- `POST /api/projects/<project_id>/sprite-model/update`
- `POST /api/projects/<project_id>/sprite-model/recover-occlusion`
- `POST /api/projects/<project_id>/rig/build`
- `POST /api/projects/<project_id>/clips/<clip_name>/render`

Retain existing:

- concept generation
- refinement
- QA
- export

`sprite-model/update` must support deterministic mutation operations only.

Use an operation payload like:

```json
{
  "operation": "set_pivot",
  "part_name": "upper_arm_right",
  "pivot_point": [12, 8]
}
```

Supported operations:

- `set_pivot`
- `set_draw_order`
- `set_parent_joint`
- `rename_part`
- `merge_parts`
- `split_part`
- `translate_part`
- `rotate_part`
- `scale_part`
- `replace_mask`
- `cleanup_alpha`
- `normalize_outline`
- `apply_palette_change`

## Implementation Order

### Milestone 1: Honest Production Boundary

- remove placeholder language implying the concept already becomes the final sprite
- add `Master Pose` stage to wizard and workbench
- gate production on `approved_master_pose`

### Milestone 2: Master Pose And Sprite Model

- implement master-pose generation
- implement sprite-model build
- persist extracted parts
- add sprite-model correction UI

### Milestone 3: Real Rig And Real Render

- replace placeholder layer source with `SpriteModel`
- replace placeholder rig build with rig-from-model
- render real deterministic idle/walk clips from extracted parts

### Milestone 4: QA And Export Hardening

- strengthen QA around deterministic assets
- export only from real sprite model path
- keep legacy placeholder pipeline available only as debug/legacy mode

### Milestone 5: Additional Clips

- add jump
- add attack
- add corrective assets where required

## Explicit Legacy Rule

The existing procedural part library may remain temporarily only if all of the following are true:

- it is clearly marked legacy/debug
- it is not the default production path
- export for production-ready assets requires the real sprite-model path

Do not present the procedural part library as production-ready once this work begins.

## Acceptance Criteria

This implementation is complete only when all of these are true:

- two visibly different approved concepts produce two visibly different final sprites
- the approved concept leads to an approved master pose
- the approved master pose leads to a real extracted sprite model
- the user can inspect and correct extracted parts in-tool
- the rig is built from extracted parts, not synthetic placeholders
- final idle and walk frames are rendered without AI calls
- rebuilding the same approved sprite model produces identical output
- export is blocked on QA failure
- every wizard step shows the correct current visualization for its state
- a non-technical user can see what the system is doing and what state the sprite is in at every step

## Definition Of Done

Done means:

- concept chosen
- master pose approved
- sprite model extracted and corrected
- rig approved
- deterministic clips rendered
- QA passed
- export produced

Done does not mean:

- concept board looks good but final sprite still comes from the hardcoded part library
