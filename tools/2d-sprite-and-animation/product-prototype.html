Positioning

Sprite Workbench is for building a single side-view character all the way through to game-ready output. The value is continuity: one brief, one look, one motion pass, one clean handoff.

The workflow stays simple on purpose: define the character, choose the look, animate it, export the package.

Who it fits.

Best when the job is specific and the handoff matters.

Best for

  • Solo devs and small teams building side-view 2D games
  • Projects that need a direct line from character concept to runtime assets
  • Teams that want local project ownership instead of cloud-only state

Not for

  • Broad concept exploration across many styles
  • Front-view, isometric, or full-sheet character pipelines
  • Large collaboration suites or marketplace workflows

Local-first by design.

Project export and runtime export stay separate because they solve different problems.

Ownership

The editable project belongs to you. The runtime package belongs to the game. Keeping those exports separate makes the workflow easier to trust, easier to recover, and easier to move between machines.

Import Project Restore a local project bundle into the workbench.
Export Project Preserve editable project state for transfer, backup, or recovery.
Export Package Produce the runtime-facing bundle with sheets, metadata, and previews.

Quickstart

  1. Open Sprite Creation and create a project.
  2. Describe the character in plain language and save the setup.
  3. Create or import concepts, then lock one side-view source.
  4. Generate motion clips in the Animations step.
  5. Run checks and export the runtime package in Review & Export.
The intended output is a side-view character package you can carry into runtime.

Workflow overview

  • Describe: capture role, silhouette, materials, prop, and tone.
  • Concepts: generate or import options, then lock the side-view source you want to keep.
  • Animations: build named motion clips from that locked source.
  • Review & Export: check the package and ship the runtime outputs.

Export format

The workbench separates editable project preservation from runtime output.

  • Project export: a restorable project bundle for backup and transfer.
  • Runtime export: the package the game consumes.

The current runtime package includes:

  • animation_sheets/<clip>.png
  • animation_sheets/<clip>.json
  • atlas.json and animations.json
  • Preview GIFs for quick review
  • Export manifest and verification output

Phaser guide

Ashen Hollow already makes Phaser the clearest target for the current runtime package.

  • Use the per-animation sheets and metadata as the first-class runtime source.
  • Map locomotion states to clips like idle, run, and jump.
  • Treat action states like attack and parry as deliberate runtime triggers.
  • Expect to tune scale, origin, and collision fit once the sprite is in-engine.

FAQ

Who is this for?
Small teams and solo devs building side-view 2D characters.

Why local-first?
So projects can be restored, moved, and backed up without treating the browser as the only home for the work.

Why separate Export Project from Export Package?
One preserves the editable project. The other produces the runtime artifact.

Is this meant to be a general-purpose sprite editor?
No. The current product direction is intentionally narrower than that.
